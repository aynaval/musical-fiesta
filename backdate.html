The Swarm Tax

What six agents cost me that one agent wouldn't have

The demo went perfectly. They usually do.

Six agents. An orchestrator routing between them. A ReAct loop reasoning over requirements data served through MCP. A Slack message that landed in the client's channel before anyone in the room had finished saying "wow." Somebody used the word transformative. I went home feeling like an architect.

Then we ran it against real data for a week.

Nothing broke, exactly — and that's the insidious part. The system didn't crash. It drifted. An agent three hops downstream would produce something confidently wrong because an agent two hops upstream had summarised away the one field that mattered. Latency crept. Token spend crept faster. And when a run went sideways, finding out which agent did that took me longer than doing the task by hand would have.

I've come to think of what I was paying as a swarm tax. And I think most teams building multi-agent systems right now are paying it without ever reading the bill.

Why we all build the swarm anyway

The multi-agent pitch is seductive because it flatters something we already believe: that complex work gets done by teams.

So the architecture diagram writes itself. A planner agent. A retrieval agent. A validation agent. An escalation agent. It maps cleanly onto an org chart, which makes it easy to explain to a stakeholder, which means it gets approved.

I have never once had to defend a multi-agent design to a client. I have had to defend a single-agent design constantly. "Only one agent? Is that… enough?"

That asymmetry should worry us. We are choosing architectures because they demo well and diagram well, not because they evaluate well.

And this isn't a niche mistake. Multi-agent designs now account for roughly two-thirds of enterprise agentic deployments. Meanwhile, only about 31% of organisations have any agent running in production — even though 80% of enterprise applications shipped last quarter arrive with one embedded.

There is a very large gap between the number of agents being designed and the number of agents that survive contact with production. I don't think that's a coincidence.

The four bills

Every agent boundary you draw is a purchase. It buys you specialisation. Here is what it charges you.

Bill 1 — Context fragmentation

The Cognition team has been making this argument for a while, and they're right: agents in a swarm do not share a global context. Every handoff is a lossy compression step. Agent A read the full document. Agent B gets Agent A's summary of the document. Agent C gets a summary of a summary.

Three hops in, your system has quietly played a game of telephone with your client's data.

The failure this produces gets logged as hallucination, and then somebody spends two weeks tuning prompts. But it is not a hallucination problem. It is an amnesia problem. The model didn't invent anything — it acted correctly on a context that had been eroded upstream.

I've watched teams, mine included, burn sprints on the wrong diagnosis, because "the model hallucinated" is a far more comfortable story than "our architecture destroys information on purpose."

Bill 2 — Token multiplication

Handoffs are not free. Each one re-establishes context: the orchestrator pays tokens to summarise, the specialist pays tokens to re-read, and the result travels back up the chain to be summarised again.

Cost does not scale with the number of agents. It scales with the number of handoffs — and handoffs have an unpleasant habit of scaling quadratically the moment you allow any agent-to-agent chatter that isn't a strict pipeline.

The research is catching up to this. A single agent with an adequate thinking budget frequently delivers more reliable multi-hop reasoning, more cheaply, than a swarm splitting the same problem across partial contexts. And most published swarm-vs-single comparisons are confounded: the swarm quietly got more test-time compute. Give the single agent the same budget and a lot of the magic evaporates.

Bill 3 — A non-deterministic failure surface, times n

One agent is one place where a language model can make a novel decision. Six agents are six such places, plus the orchestrator's routing decision, plus every retry.

The cost shows up at 2am, not in the design review.

Retry semantics across a handoff are genuinely hard. If Agent D fails, do you retry D with the same input? Re-run C to produce a fresh input? Restart the whole trajectory? Every answer is wrong in some scenario. Meanwhile your traces are a forest rather than a line, and root-causing one bad output means reading six reasoning chains to find where the drift began.

Bill 4 — The eval explosion (the one nobody prices in)

This is the bill that changed how I build.

With one agent, you have one trajectory to evaluate. With six, you have at least six — plus the routing decisions between them, plus the emergent behaviour of the whole thing.

Evaluation-driven development — writing your eval criteria before you write agent code, TDD-style — is now table stakes for anything going to production. The working guidance is that you need on the order of 500 cases before an aggregate metric means anything at all.

Multiply that by your agent count, then be honest with yourself about whether you're actually going to do it.

Most teams don't. They evaluate the final output, ship, and discover months later that Agent 4 has been quietly wrong 8% of the time in a way that Agents 5 and 6 were papering over.

If you can't afford to evaluate an agent, you can't afford to have it.

That sentence would have saved me a quarter.

Where the boundary genuinely earned its keep

I'm not arguing multi-agent is a fashion. In my own build, three boundaries paid for themselves — and what they had in common is that none of them were about roles.

Different blast radius. The agent that reads untrusted external content should not be the agent holding write credentials to a system of record. This has stopped being theoretical. We've now seen malicious MCP servers caught in the wild — one shipped fifteen clean versions to build trust, then quietly added a single line of exfiltration code — alongside critical RCEs in core MCP infrastructure, a widely-used LLM gateway backdoored on a package registry, and tens of thousands of live secrets sitting in MCP config files on public GitHub.

A boundary that constrains what a compromised agent can reach is a security control, not an architectural flourish. This is now, by a distance, the best reason to split an agent.

Genuinely parallel, genuinely independent work. If two subtasks don't need each other's intermediate state and you're latency-bound, split them. If they do need each other's state — congratulations, you've just bought Bill 1.

Context that truly does not fit. Not "the prompt is getting long." That's a context engineering problem, and I'll come back to it. I mean a tool surface and a domain vocabulary so different that keeping them together measurably degrades both.

Notice what isn't on that list: "because the workflow has four steps."

Steps are not agents. Steps are steps.

The checklist I'd hand my past self

Before something gets its own agent, make it answer yes to at least one:

  1. Blast radius — does it need a different permission scope or trust boundary than everything around it?
  2. Parallelism — can it run concurrently with genuinely independent state, and am I actually latency-bound?
  3. Context incompatibility — does keeping it in the main context measurably degrade the main task? Measurably. As in, you measured it.
  4. Independent lifecycle — will it be owned, versioned and deployed by a different team on a different cadence?
  5. Evaluability — can I write and maintain a real eval suite for it, today, at my current headcount?

If the answer to all five is no, what you have is not an agent. It's a tool, a skill, or a branch in a flow.

Ship it as one.

What I'd build by default now

One agent. A tight tool surface. And context engineering as the first lever I reach for — not architecture.

The industry has quietly converged here while everyone was arguing about frameworks. Thoughtworks' Technology Radar now lists context engineering as Adopt, progressive context disclosure as Trial, and — my favourite entry on the whole radar — agent instruction bloat as Caution.

The pattern that made the difference for me is progressive disclosure. The agent sees only the names and descriptions of its capabilities at rest — roughly 80 tokens each — and loads the full instruction body only once it decides that capability is relevant. Scripts and reference material load later still, only during execution.

That single move gives you most of what people are hoping to get from a swarm. Real specialisation — each skill has its own instructions, its own scripts, its own reference docs — without context fragmentation, without the handoff tax, and with exactly one trajectory to evaluate.

Most of the "agents" I've built could have been skills.

That is an uncomfortable thing to write. It is also why I'm writing it.

The part that actually matters

Enterprise AI is moving from build an agent to operate an agent estate. Every major vendor is now selling a control plane for that estate — governance, policy enforcement, auditability across thousands of agents from any source. It's real, it's necessary, and it's the layer I work on for a living.

But there's a step that keeps getting skipped, and it's cheaper than every tool you'll buy to solve it.

Most of the estate shouldn't exist.

Agent count is not a measure of sophistication. It's a measure of how much complexity you have committed to operating, evaluating, securing and paying for — indefinitely.

Before you add the seventh agent, make it justify the tax.

Build less. Evaluate more. Your 2am self will thank you.
